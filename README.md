# docxform — GUI-шаблонизатор документов .docx (Qt5)

`docxform` открывает `.docx` с плейсхолдерами `\var{имя}`, тегами таблиц и
текстовыми тегами, показывает форму с полем ввода на каждую переменную и
сохраняет готовый `.docx` с подставленными значениями.

## Как это работает

Графическая программа на C++ / Qt5. Работает по принципу Jinja:

1. В Word/Google Docs/LibreOffice вы пишете документ и помечаете изменяемые
   места плейсхолдерами `\var{имя}` (имя — латиницей), напр.:
   «Уважаемый `\var{client_name}`, ваш заказ `\var{order_id}` — `\var{status}`.»
2. `docxform` открывает `.docx`, находит все `\var{...}` и показывает окно с
   полем ввода на каждую переменную (подпись поля = имя переменной).
3. Вы вводите значения и нажимаете «Создать документ…».
4. На выходе — новый `.docx`, где каждый `\var{...}` заменён на введённое
   значение, а сами подставленные значения **выделены жёлтым** (выделение можно
   отключить — см. ниже).

### Особенности

- **Выделение вставленного текста.** В окне есть флажок «Выделять вставленный
  текст жёлтым» (включён по умолчанию). Если его снять, подставленные значения
  вставляются без жёлтого выделения. В headless-режиме за это отвечает флаг
  `--no-highlight`.

- **Группировка по абзацам и порядок документа.** Абзац документа показывается
  один раз (целиком — и текст до плейсхолдеров, и после), а под ним идут
  элементы управления для его плейсхолдеров. Переменные, варианты и таблицы
  идут **в том же порядке, что и в документе** (вперемешку, а не «сначала все
  переменные, потом все таблицы»). В тексте абзаца активные плейсхолдеры
  подсвечены, остальные — серым. Каждый плейсхолдер относится к абзацу своего
  первого вхождения и в форме встречается один раз.
- **Подсказки.** У каждого поля есть tooltip с подробным описанием.
- **Разорванные плейсхолдеры.** Плейсхолдеры, разбитые редактором на несколько
  фрагментов (частое явление в Word), собираются и заменяются корректно.
- **Сохранение форматирования.** Оформление подставляемого места (полужирность,
  размер и т.п.) берётся из исходного плейсхолдера; к нему добавляется жёлтое
  выделение.
- Картинки, таблицы, табуляции и прочее оформление сохраняются без потерь;
  плейсхолдеры внутри таблиц тоже обрабатываются.
- Имя переменной: буквы/цифры/подчёркивание (`[A-Za-z0-9_]`); вокруг имени
  внутри скобок допускаются пробелы: `\var{ order_id }`.

## Переменные: `\var{имя}` и `\var{имя|подсказка}`

Переменная задаётся плейсхолдером `\var{...}` в двух формах:

- `\var{имя}` — простая переменная. `имя` — латиницей, `[A-Za-z0-9_]`: это и
  подпись поля в форме, и ключ для подстановки. Поле получает **стандартную**
  подсказку (tooltip). Если имя встречается несколько раз, поле для него
  показывается **один раз**, а введённое значение подставляется во все вхождения.
- `\var{имя|подсказка}` — то же самое, но текст `подсказка` (всё, что стоит
  **после первой вертикальной черты `|`**) показывается в качестве tooltip’а поля
  **вместо** стандартной подсказки. Разделитель `|` — тот же, что и у
  `\variant{...}`. `подсказка` необязательна; это произвольный текст: можно с
  пробелами, пунктуацией, запятыми, `|` и на русском (имя берётся до первой `|`).
  Например:
  - `\var{inn|Введите ИНН (10 или 12 цифр)}` → поле `inn`, tooltip
    «Введите ИНН (10 или 12 цифр)».
  - `\var{total|Сумма, в рублях}` → поле `total`, tooltip «Сумма, в рублях»
    (запятые внутри подсказки не мешают).

В обеих формах в итоговом документе на месте `\var{...}` оказывается то, что
пользователь ввёл в поле (выделенное жёлтым). Если поле оставить пустым,
плейсхолдер удаляется из документа.

Замечания:

- Как и раньше, переменная с одним и тем же именем встречается в форме **один
  раз** и подставляется во все свои вхождения.
- Подсказку `info` можно указать на **любом** вхождении имени — например, в одном
  месте написать `\var{DoxNum}`, а в другом `\var{DoxNum|Номер документа}`: поле
  всё равно получит подсказку «Номер документа». Если подсказка задана у
  нескольких вхождений, берётся **первая по тексту** документа.
- Разорванные редактором `\var{...}` (частое явление в Word) собираются и
  обрабатываются корректно.
- Некорректный плейсхолдер (пустое или невалидное имя, напр. `\var{}`,
  `\var{имя с пробелом}`) не распознаётся и остаётся в тексте как есть.
- В headless-режиме (`--render`) не заданная переменная остаётся в документе в
  виде исходного литерала `\var{...}`.

## Таблицы: теги `\tablewage`, `\tableprice`, …

Кроме переменных, в шаблоне можно вставлять таблицы. У **каждого вида таблицы
свой тег без скобок**, который **всегда** подставляет одну заданную таблицу:
никакого выбора в окне и аргументов в headless — тег вставляется
**автоматически**, выполняя привязанную к нему функцию-логику.

1. В нужном месте документа пишете тег, напр. `\tablewage` (лучше в **отдельном
   абзаце** — весь абзац с тегом заменяется на таблицу).
2. При генерации на месте тега появляется готовая таблица с рамками и
   выравниванием текста по центру; содержимое возвращает привязанная функция.

Сами таблицы (сколько строк/столбцов, имена столбцов и содержимое) задаются в
исходном коде — в файле `tablekinds.cpp`. Содержимое может быть как
фиксированным, так и вычисляться в момент генерации (см. демо «График платежей»
и «строки из данных программы» ниже).

### Как добавить свой тег

1. Напишите функцию-builder с логикой таблицы (в `tablekinds.cpp`): фиксированные
   `headers`, а `rows` — как угодно (константа, вычисление, данные программы).
   Готовые примеры: `wageReport` (своя логика под тег), `runtimeTable` (строки из
   `tableContext()`), `priceList`, `employees`, `paymentSchedule`, `blankGrid`.
2. В функции `fixedTables()` добавьте строку `{тег, функция}`:

   ```cpp
   std::vector<FixedTable> fixedTables() {
       std::vector<FixedTable> fixed;
       fixed.push_back({"\\tableprice",   priceList});     // готовый builder
       fixed.push_back({"\\tablewage",    wageReport});    // своя логика под тег
       fixed.push_back({"\\tableruntime", runtimeTable});  // строки из tableContext()
       return fixed;
   }
   ```

   Сигнатура builder’а: `TableData f(const std::string& tag)` — на вход приходит
   сам тег (одна функция может обслуживать несколько тегов, различая их по `tag`).

3. Готово: написав тег в документе, вы всегда получите его таблицу.

Требования к тегам: начинаются с `\`, состоят из латиницы/цифр (`\tableprice`),
различны и не являются префиксом друг друга. Совпадение учитывает границу слова,
поэтому `\tablewage` не сработает внутри `\tablewages`. Список распознанных тегов
в документе можно увидеть командой `./docxform --vars файл.docx` (строки
`FIXEDTABLE`).

В комплекте есть демо-теги: `\tableblank`, `\tableprice`, `\tableemployees`,
`\tableschedule`, `\tablewage`, `\tableruntime`, `\tablespan` (замените на свои).

### Объединённые ячейки / разное число колонок в строках

Если у строк **разное число колонок** (объединённые ячейки), задайте в `TableData`
**спаны** — сколько колонок сетки занимает каждая ячейка (поля `headerSpans` /
`rowSpans`, параллельные `headers` / `rows`; пропуск или `<= 0` = 1). Сетка
получает столько колонок, сколько в самой «широкой» строке по сумме спанов.

Например, первая строка из 3 ячеек над второй из 5 (2-я ячейка над колонками
2-3, 3-я — над 4-5):

```cpp
TableData spanDemo(const std::string&) {
    TableData t;
    t.headers = {"Группа A", "Группа B", "Группа C"};
    t.headerSpans = {1, 2, 2};               // 1 + 2 + 2 = 5 колонок сетки
    t.rows = { {"a", "b1", "b2", "c1", "c2"} };  // 5 ячеек, спаны по умолчанию = 1
    return t;
}
```

Реализуется через OOXML `<w:gridSpan>`; обычные равноколоночные таблицы не
меняются (спаны не заданы → все ячейки по 1 колонке). Рабочий пример — демо-тег
`\tablespan`.

**Итог по вашим вопросам:**

- **Разное число столбцов в строках** — да: в каждой строке кладёте сколько нужно
  ячеек и задаёте им спаны (все строки делят одну сетку — это требование Word;
  границы 3-колоночной строки должны попадать на границы 5-колоночной).
- **Ширина столбцов** — настраивается полем `colWidths` (относительные веса
  колонок сетки; пусто → все равные). Ячейка со спаном получает суммарную ширину
  своих колонок. Пример: первая колонка вдвое шире остальных —

  ```cpp
  t.colWidths = {2, 1, 1, 1, 1};  // 5 колонок: первая в 2 раза шире
  ```

Таблица всегда занимает одну и ту же общую ширину страницы; веса лишь делят её.

### Наполнение таблицы данными программы (в рантайме)

Содержимое таблицы вычисляется в момент генерации: функция `build()` у вида
таблицы вызывается каждый раз заново и возвращает свежий `TableData`. Поэтому
**колонки можно держать фиксированными, а строки/ячейки — генерировать на лету**,
в том числе из данных, которые ваша программа передаёт в рантайме.

Для передачи данных из встраивающего приложения есть `tableContext()` —
process-wide хранилище «ключ → строки таблицы». Шаблон использования:

1. В `tablekinds.cpp` напишите `build()`, который держит фиксированные `headers`,
   а `rows` берёт из `tableContext()` по имени (`name` — это имя тега, напр.
   `\tablewage`). Готовый демо-вид — `runtimeTable` (id `runtime`), на него
   указывает фиксированный тег `\tableruntime`:

   ```cpp
   TableData runtimeTable(const std::string& name) {
       TableData t;
       t.headers = {"Колонка A", "Колонка B"};   // колонки фиксированы
       auto it = tableContext().find(name);
       if (it != tableContext().end()) t.rows = it->second;  // строки из программы
       return t;
   }
   ```

2. В своей программе **перед генерацией** заполните данные (ключ — тот же тег):

   ```cpp
   docxform::tableContext()["\\tablewage"] = {
       {"Иванов И. И.", "100000"},
       {"Петров П. П.", "90000"},
   };
   docxform::showTemplateForm(this);   // дальше — обычный сценарий
   ```

3. При генерации на месте тега появится таблица с вашими строками и
   неизменными колонками.

Тип значения в `tableContext()` (по умолчанию — строки таблицы
`vector<vector<string>>`) можно поменять в `tablekinds.h` под свои данные. Если
данные удобнее тянуть прямо в `build()` (из файла, БД, других глобалей вашего
приложения) — `tableContext()` не обязателен: `build()` исполняется как обычный
C++-код и может взять данные откуда угодно.

## Текстовые теги: `\company`, `\today`, …

Кроме таблиц, можно завести **свои теги для обычного текста**: бара-тег без
скобок, который при генерации заменяется **inline** на текст, возвращаемый вашей
функцией (в отличие от тега таблицы, который заменяет весь абзац). Тег ведёт себя
как заданный в коде `\var`: подставляется в любом месте предложения, сохраняя
текст и оформление вокруг; ввод пользователя и аргументы не нужны.

Как добавить свой тег:

1. Напишите функцию, возвращающую `std::string` (константу или вычисление), в
   `tablekinds.cpp`. Готовые примеры: `companyText` (константа), `todayText`
   (текущая дата).
2. В функции `fixedTexts()` добавьте строку `{тег, функция}`:

   ```cpp
   std::vector<FixedText> fixedTexts() {
       std::vector<FixedText> texts;
       texts.push_back({"\\company", companyText});  // константный текст
       texts.push_back({"\\today",   todayText});    // вычисляется при генерации
       return texts;
   }
   ```

   Сигнатура: `std::string f(const std::string& tag)`. Возвращаемый текст —
   обычный (он экранируется при вставке), поэтому вставить «сырой» OOXML нельзя.

3. Теперь, написав в документе `\company`, вы всегда получите соответствующий
   текст.

Правила те же, что и для тегов таблиц: тег начинается с `\`, теги различны и не
являются префиксом друг друга (в т.ч. между текстовыми тегами и тегами таблиц).
Совпадение учитывает границу слова: `\company` не сработает внутри
`\companywide`. Подставленный текст **выделяется жёлтым** так же, как значения
переменных, — по флажку «Выделять вставленный текст жёлтым» (в headless — по
`--no-highlight`). Список распознанных тегов: `./docxform --vars файл.docx`
(строки `FIXEDTEXT`).

## Варианты: `\variant{имя|текст 1|текст 2|…}`

Когда в одном месте нужно дать на выбор несколько готовых формулировок,
используйте вариант:

1. В нужном месте пишете `\variant{имя|текст 1|текст 2|…}`. **Имя** — это часть
   до первого `|`; оно может быть **на русском и из нескольких слов** (например,
   `обращение к клиенту`). **Варианты** — это части, разделённые `|`, после
   имени.
2. При запуске `docxform` находит все варианты и показывает рядом с каждым
   **выпадающий список** с предложенными формулировками (подпись = имя).
3. Вы выбираете нужный вариант.
4. На выходе на месте `\variant{…}` остаётся выбранный текст (выделенный
   жёлтым).

Особенности:

- Варианты берутся **дословно** (пробелы и пунктуация сохраняются), поэтому их
  удобно использовать и для подстановок внутри слова, напр.
  `Уважаем\variant{род|ый|ая}` → «Уважаемый» / «Уважаемая».
- Вариант может быть **пустым** (`\variant{прим| (важно)|}`) — выбор пустого
  варианта просто ничего не вставит.
- Как и переменные, разорванные редактором `\variant{…}` собираются и
  обрабатываются корректно.
- Если вариант не выбран в headless-режиме, берётся **первый** из списка.

**Модульность:** всё, что касается таблиц (структуры данных, реестр тегов и
рендеринг OOXML), вынесено в `tablekinds.h` / `tablekinds.cpp` и не зависит от
остального `docxform` — этот модуль можно переиспользовать в другом
Qt-приложении (`fixedTables()` для списка тегов и `buildTableXml()` для
рендеринга).

## Требования

- Компилятор C++ с поддержкой C++17.
- Qt5 (модуль Widgets) и заголовки к нему.
  - Ubuntu/Debian: `sudo apt install qtbase5-dev`
  - Fedora: `sudo dnf install qt5-qtbase-devel`
- `pkg-config` — нужен для сборки через Makefile (читает Qt-флаги).
  - Ubuntu/Debian: `sudo apt install pkg-config`
  - Fedora: `sudo dnf install pkgconf-pkg-config`
- Библиотека zlib и заголовки к ней.
  - Ubuntu/Debian: `sudo apt install zlib1g-dev`
  - Fedora: `sudo dnf install zlib-devel`

Проверить Qt: `pkg-config --modversion Qt5Widgets`

## Сборка

```sh
make docxform
# или вручную:
g++ -O2 -std=c++17 -fPIC docxform.cpp tablekinds.cpp -o docxform \
    $(pkg-config --cflags --libs Qt5Widgets) -lz
```

## Кроссплатформенность (Linux / Windows / macOS)

Код использует только стандартный C++17, Qt5 и zlib — никаких ОС-специфичных
заголовков. Разбор/сборка ZIP (`.docx`) сделаны ручными побайтовыми операциями,
поэтому **не зависят от порядка байт** (работают и на little-, и на big-endian).
Файловый ввод/вывод идёт через `QFile`, поэтому **пути с не-ASCII символами
(кириллица и т.п.) корректно открываются и на Windows**.

- **Linux / macOS:** `make` (Linux) или qmake; компилируется как gcc, так и
  clang; zlib линкуется через `-lz`.
- **Windows:** собирайте через qmake — подключите `docxform.pri` в свой `.pro`
  (см. «Встраивание»). Поддерживаются оба варианта **clang** (а также MSVC):
  - **clang в MinGW-режиме** (`win32-clang-g++`) и **MinGW gcc** — zlib
    подхватывается через `-lz` автоматически;
  - **clang-cl** (`win32-clang-msvc`) и **MSVC** — стандартного zlib нет, укажите
    свой (пример с путями в комментарии внутри `docxform.pri`).

  `.pri` сам выбирает нужный вариант по типу драйвера (qmake-scope `gcc`).
  `Makefile` рассчитан только на Linux — на Windows используйте
  qmake/`docxform.pri` или свою сборку.

Замечание: для **CLI** на Windows пути берутся из узкого `argv`, поэтому имена с
не-ASCII символами в командной строке зависят от кодовой страницы консоли; в
GUI-сценарии (через диалоги Qt) таких ограничений нет.

## Запуск

```sh
./docxform                 # откроется диалог выбора .docx
./docxform шаблон.docx     # сразу открыть указанный файл
```

Без графики (для скриптов/проверки) есть «тихий» режим — он делает то же
самое, но без окна; значения берутся из аргументов:

```sh
./docxform --render <вход.docx> <выход.docx> \
    ИМЯ=ЗНАЧЕНИЕ ... [~ВАРИАНТ=выбор ...] [--no-highlight]
```

- `ИМЯ=ЗНАЧЕНИЕ` — значение переменной `\var{ИМЯ}` (для формы
  `\var{ИМЯ|подсказка}` ключ — это `ИМЯ`, т.е. часть до первой `|`).
- `~ИМЯ=выбор` — выбор для `\variant{ИМЯ|…}`. `выбор` — это **номер** варианта
  (с 1) или его **текст** дословно. Имена с пробелами берите в кавычки.
- `--no-highlight` — вставлять значения **без** жёлтого выделения (по умолчанию
  выделяются).

Теги таблиц (`\tablewage`, …) подставляются **автоматически** — аргументы для них
не нужны.

Пример:

```sh
./docxform --render in.docx out.docx "client=ООО Ромашка" \
    "~род=2" "~обращение к клиенту=Уважаемые партнёры"
```

Посмотреть найденные плейсхолдеры (переменные, варианты и теги таблиц) в порядке
документа — то, что покажет GUI:

```sh
./docxform --vars <вход.docx>
```

Формат вывода `--vars` (через табуляцию):

```
VAR         <имя>   <контекст-абзац>
VARIANT     <имя>   <вариант1>|<вариант2>|…
FIXEDTABLE  <тег>
FIXEDTEXT   <тег>
```

## Встраивание окна в другую программу

Окно `docxform` можно открыть прямо из своего Qt-приложения, не запуская
отдельный бинарник. Публичный интерфейс объявлен в `docxform.h`:

```cpp
namespace docxform {
    // Полный «обособленный» сценарий одним вызовом: показывает диалог выбора
    // .docx, строит форму и показывает её как окно (удаляется при закрытии);
    // об ошибке сообщает сам. Возвращает показанное окно или nullptr, если
    // пользователь отменил выбор файла либо открытие не удалось. Это ровно то,
    // что делает обычный запуск без аргумента — удобно вешать на кнопку.
    QWidget* showTemplateForm(QWidget* parent = nullptr);

    // Строит окно-форму для УЖЕ известного шаблона .docx и возвращает его как
    // QWidget. Владелец виджета — вызывающий код (показать через show(), при
    // желании задать родителя). При ошибке возвращает nullptr и кладёт текст
    // ошибки в *error; сам диалогов не показывает. QApplication к моменту
    // вызова уже должен существовать.
    QWidget* openTemplateForm(const QString& templatePath,
                              QString* error = nullptr,
                              QWidget* parent = nullptr);
}
```

### Запуск модуля по нажатию кнопки (как при обособленном запуске)

Чтобы по кнопке в вашей форме открывался весь сценарий docxform (выбор файла →
форма → сохранение), достаточно одного вызова `showTemplateForm()`:

```cpp
#include "docxform.h"
...
connect(myButton, &QPushButton::clicked, this, [this] {
    docxform::showTemplateForm(this);  // this — родитель окна (можно nullptr)
});
```

`showTemplateForm()` сам покажет диалог выбора `.docx`, построит форму, отобразит
её и при ошибке покажет сообщение. Если хотите свой диалог выбора файла —
выбирайте путь сами и вызывайте `openTemplateForm()`:

```cpp
QString err;
if (QWidget* form = docxform::openTemplateForm("contract.docx", &err)) {
    form->setAttribute(Qt::WA_DeleteOnClose);  // удалится при закрытии
    form->show();
} else {
    QMessageBox::critical(this, "Ошибка", err);
}
```

Окно само спросит значения переменных и выпадающие списки для `\variant{…}`, а
по кнопке «Создать документ…» запишет готовый `.docx` (с диалогом сохранения;
теги таблиц и текста подставятся автоматически — данные для них см. ниже в
разделе «Подача данных таблиц и текста из вашей программы»). Дальше вмешательство
хост-программы не требуется.

Сборка вместе со своим приложением: компилируйте `docxform.cpp` с макросом
`DOCXFORM_NO_MAIN` (он убирает встроенный `main()`, чтобы не конфликтовать с
вашим) и слинкуйте с `tablekinds.cpp`:

```sh
g++ -O2 -std=c++17 -fPIC -DDOCXFORM_NO_MAIN \
    моя_программа.cpp docxform.cpp tablekinds.cpp \
    $(pkg-config --cflags --libs Qt5Widgets) -lz
```

Без `DOCXFORM_NO_MAIN` `docxform.cpp` собирается как самостоятельная программа
(встроенный `main` с GUI и режимами `--render` / `--vars`). Набор доступных
видов таблиц настраивается, как обычно, в `tablekinds.cpp`.

### Подключение в qmake-проект (3rd-party)

В комплекте есть файл `docxform.pri`. Скопируйте каталог `docxform` к себе
(например, в `third_party/docxform`) и в своём `.pro` подключите один файл:

```pro
include(third_party/docxform/docxform.pri)
```

`docxform.pri` сам добавит исходники и заголовки (`docxform.cpp`,
`tablekinds.cpp`, `docxform.h`, `tablekinds.h`), пропишет `QT += widgets`,
`INCLUDEPATH`, слинкует zlib (`LIBS += -lz`) и определит `DEFINES +=
DOCXFORM_NO_MAIN`, чтобы убрать встроенный `main()` (ваш `main` остаётся
вашим). Пути внутри `.pri` заданы через `$$PWD`, так что include работает из
любого места.

После этого вызывайте `docxform::openTemplateForm()` (см. пример выше);
`#include "docxform.h"` найдётся через `INCLUDEPATH`.

Минимальный пример `.pro`:

```pro
TEMPLATE = app
TARGET   = myapp
CONFIG  += c++17
SOURCES += main.cpp
include(third_party/docxform/docxform.pri)
```

Требования те же, что и для standalone-сборки: Qt5 Widgets и zlib. Если ваш
проект собирает несколько целей или использует другие пути — `.pri` это не
меняет, он только дописывает свои `SOURCES`/`HEADERS`/`LIBS` к текущему
проекту.

### Подача данных таблиц и текста из вашей программы

Это основной сценарий интеграции: ваш модуль **генерирует данные**, а docxform
подставляет их в документ. Откуда что берётся:

| Что в документе | Кто заполняет | Как |
|---|---|---|
| `\var{имя}` | **пользователь** в форме | поле ввода |
| `\variant{…}` | **пользователь** в форме | выпадающий список |
| тег таблицы (`\tableorders`) | **ваша программа** | `tableContext()` |
| текстовый тег (`\client`) | **ваша программа** | `textContext()` |

Поток данных: вы **до генерации** кладёте свои данные в `tableContext()` /
`textContext()`, а привязанные к тегам builder-функции читают их **в момент**
нажатия «Создать документ». То есть колонки/структура заданы в коде, а конкретные
значения приходят из вашей программы в рантайме.

**Шаг 1 — один раз привяжите свои теги к билдерам, читающим контексты**
(в `tablekinds.cpp`; готовые демо `runtimeTable` и `contextText` уже так делают):

```cpp
// Тег таблицы \tableorders: колонки фиксированы, строки — из вашей программы.
TableData buildOrders(const std::string& tag) {
    TableData t;
    t.headers = {"Товар", "Кол-во", "Цена"};        // структура задаётся здесь
    auto it = tableContext().find(tag);             // tag == "\\tableorders"
    if (it != tableContext().end()) t.rows = it->second;
    return t;
}

// Текстовый тег \client: значение — из вашей программы.
std::string buildClient(const std::string& tag) {
    auto it = textContext().find(tag);
    return it != textContext().end() ? it->second : std::string();
}

std::vector<FixedTable> fixedTables() {
    std::vector<FixedTable> v;
    v.push_back({"\\tableorders", buildOrders});
    return v;
}
std::vector<FixedText> fixedTexts() {
    std::vector<FixedText> v;
    v.push_back({"\\client", buildClient});
    return v;
}
```

**Шаг 2 — в вашем модуле положите сгенерированные данные и откройте форму:**

```cpp
#include "docxform.h"
#include "tablekinds.h"

void MyWidget::onFillButtonClicked() {
    // данные, которые сгенерировала ваша программа:
    docxform::tableContext()["\\tableorders"] = {
        {"Стол", "2", "5000"},
        {"Стул", "6", "1500"},
    };
    docxform::textContext()["\\client"] = "ООО «Ромашка»";

    // открыть сценарий docxform (выбор файла → форма → сохранение):
    docxform::showTemplateForm(this);
}
```

Когда пользователь заполнит `\var`/`\variant` и нажмёт «Создать документ…», в
итоговом `.docx` окажутся: значения пользователя **плюс** ваши таблицы и текст из
контекстов. Заголовки `docxform.h` (форма) и `tablekinds.h` (`tableContext()` /
`textContext()` / реестры) — это весь нужный публичный API.

Замечания:

- Контексты — **process-wide**; заполняйте их **перед** открытием формы (или перед
  программной генерацией). Перегенерируете с новыми данными — просто перезапишите
  контексты.
- Тип значения в контекстах можно поменять под свои данные (в `tablekinds.h`):
  `tableContext()` хранит строки таблицы (`vector<vector<string>>`),
  `textContext()` — строки текста.
- Контексты не обязательны: builder — обычный C++-код и может брать данные откуда
  угодно (БД, файл, ваши объекты). Контексты — просто готовый удобный канал.
- Какие теги распознаны в конкретном шаблоне, видно по `./docxform --vars
  файл.docx` (строки `FIXEDTABLE` / `FIXEDTEXT`).

## Коды возврата (режим `--render`)

- `0` — успешно.
- `1` — ошибка (неверные аргументы, файл не читается / не `.docx`, или нельзя
  записать вывод).
